\section{Executive Summary}

CloudWatch AI is a fully serverless AIOps platform that predicts cloud service incidents before they occur. Built entirely on AWS, the system uses Gradient Boosting machine learning models trained on 2.4 million timesteps of cloud service KPI data to issue early warnings with \textbf{0.91 precision} and a \textbf{$\sim$36-minute average lead time}, while maintaining fewer than one false alarm per service per day.

\begin{infobox}[Key Achievements]
\begin{itemize}[leftmargin=1em, itemsep=2pt]
    \item \textbf{13 Lambda functions} implementing the full data pipeline and REST API
    \item \textbf{6 DynamoDB tables} with Streams for event-driven processing
    \item \textbf{19 JWT-protected API routes} via API Gateway
    \item \textbf{5 monitored cloud services} with real-time simulation across \textbf{6 scenarios}
    \item \textbf{3 ML models} (one per prediction horizon: 5, 10, 15 minutes)
    \item \textbf{Per-user data isolation} via Cognito-based \texttt{owner\_id} partitioning across all tables
    \item \textbf{Email alert notifications} via Amazon SNS with per-user filtering
    \item \textbf{React 19 dashboard} with 4 interactive pages, auto-polling, and dark/light theme
    \item \textbf{AWS Cognito authentication} with SRP protocol and email verification
    \item \textbf{OWASP Top 10 audit}: 9/10 passed, 1 N/A (overall risk: Low)
    \item \textbf{50+ integration test cases} across all system layers
\end{itemize}
\end{infobox}

The platform demonstrates a complete, production-grade cloud database application: from data generation and ML inference to real-time visualization and security hardening. All components run serverlessly within the AWS Free Tier in the \texttt{ca-central-1} region.

\subsection{DynamoDB Table Design}

We chose Amazon DynamoDB as our NoSQL database because each of the 5 monitored cloud services produces \textbf{a completely different JSON structure}. A relational database would require many nullable columns or separate tables per service type. DynamoDB's flexible document schema allows heterogeneous records in a single table without schema migrations.

\subsection{Per-User Data Isolation}

A key requirement was ensuring that each authenticated user sees only their own simulation data, alerts, and incidents. We implemented this by embedding the Cognito \texttt{sub} (user ID) into every table's partition key:

\begin{itemize}[leftmargin=2em]
    \item \textbf{Data tables} (KPIMetrics, ServiceLogs, FeatureStore) use a composite partition key: \texttt{owner\_service = \{owner\_id\}\#\{service\_id\}}. This ensures each user's service data is physically partitioned and queryable without cross-user leakage.
    \item \textbf{Alerts and Incidents} use \texttt{owner\_id} directly as the partition key. The Alerts table includes a GSI (\texttt{GSI\_AlertsByOwnerStatus}) on \texttt{owner\_status} (a composite of \texttt{\{owner\_id\}\#\{severity\}}) for efficient filtered queries.
    \item \textbf{SimConfig} uses \texttt{owner\_id} as the partition key, allowing concurrent independent simulations by different users.
    \item Every API Lambda extracts \texttt{owner\_id} from the JWT claims (\texttt{requestContext.authorizer.jwt.claims.sub}) and scopes all queries to that user.
\end{itemize}

\subsection{Key Design Choices}

\begin{enumerate}[leftmargin=2em]
    \item \textbf{Partition Key = \texttt{owner\_service}}: The composite key \texttt{\{owner\_id\}\#\{service\_id\}} serves as the partition key for data tables, combining per-user isolation with per-service querying in a single key. This is the most common query pattern: ``get all data for a specific service belonging to a specific user.''
    \item \textbf{Sort Key = \texttt{timestamp}}: Enables efficient time-range queries within a partition (e.g., ``last 30 minutes of KPI data for web-api-001'').
    \item \textbf{TTL (Time To Live)}: All data tables have a \texttt{ttl} attribute set to 24 hours. DynamoDB automatically deletes expired items, keeping storage costs minimal and eliminating the need for manual cleanup.
    \item \textbf{DynamoDB Streams}: Enabled on \texttt{KPIMetrics} and \texttt{FeatureStore} tables with \texttt{NEW\_IMAGE} view type. Stream events automatically trigger the Feature Engineering and Inference Lambda functions, creating a fully event-driven pipeline with zero orchestration code.
    \item \textbf{On-demand capacity}: All tables use on-demand billing, which auto-scales to handle any request volume without capacity planning---ideal for a project with variable simulation load.
\end{enumerate}

\subsection{Data Pipeline (Event-Driven)}

The core innovation of our backend is the \textbf{fully automatic, event-driven data pipeline} powered by DynamoDB Streams. No polling or orchestration code is needed. All pipeline Lambdas propagate the \texttt{owner\_id} through the chain to maintain per-user data isolation:

\begin{enumerate}[leftmargin=2em]
    \item \textbf{EventBridge} triggers the \texttt{simulator\_lambda} every 60 seconds. The simulator scans \texttt{SimConfig} for all active simulations (keyed by \texttt{owner\_id}) and generates data independently for each user.
    \item The simulator generates KPI metrics and logs for all 5 services and writes them to \texttt{KPIMetrics} and \texttt{ServiceLogs} tables, using the composite key \texttt{owner\_service = \{owner\_id\}\#\{service\_id\}}.
    \item A \textbf{DynamoDB Stream} on \texttt{KPIMetrics} automatically triggers \texttt{feature\_engineering\_lambda}, which extracts \texttt{owner\_service} and \texttt{owner\_id} from the stream record, computes 8 rolling features, and writes them to \texttt{FeatureStore} with the same composite key.
    \item A \textbf{DynamoDB Stream} on \texttt{FeatureStore} automatically triggers \texttt{inference\_lambda}, which loads 3 Gradient Boosting models from S3, runs prediction, and creates alerts in the \texttt{Alerts} table (keyed by \texttt{owner\_id} + \texttt{created\_at}) if the score exceeds the calibrated threshold.
    \item If 2+ related alerts exist for the same \texttt{owner\_id}, they are grouped into an \texttt{Incident} record.
\end{enumerate}

\subsection{Inference Lambda Implementation}

The inference function is the most complex Lambda in the system. Key implementation details:

\begin{itemize}[leftmargin=2em]
    \item \textbf{Docker deployment}: Packaged as a container image (Python 3.11 base) because scikit-learn + numpy + scipy exceed the 250 MB Lambda deployment limit. The Dockerfile:
\end{itemize}

\begin{lstlisting}[language=bash, caption={Inference Lambda Dockerfile}]
FROM --platform=linux/amd64 public.ecr.aws/lambda/python:3.11
RUN pip install "scikit-learn==1.7.2" "numpy<2.3" "scipy<1.15" \
    "joblib" "pandas" boto3 --no-cache-dir
COPY lambda_function.py ${LAMBDA_TASK_ROOT}/
CMD ["lambda_function.handler"]
\end{lstlisting}

\begin{itemize}[leftmargin=2em]
    \item \textbf{Model caching}: Models are loaded from S3 into the Lambda's global scope (\texttt{\_models} dictionary). Once loaded, they persist across warm invocations, eliminating repeated S3 downloads.
    \item \textbf{Feature extraction}: The Lambda reads the 8-feature vector from the DynamoDB Stream event in a fixed order: \texttt{roll\_mean}, \texttt{roll\_max}, \texttt{roll\_std}, \texttt{roll\_slope}, \texttt{first\_diff}, \texttt{error\_rate}, \texttt{warn\_rate}, \texttt{severity\_change\_flag}.
    \item \textbf{Prediction}: For each incoming feature vector, the Lambda runs \texttt{model.predict\_proba(X)[0, 1]} on all 3 models (horizons 5, 10, 15 minutes).
    \item \textbf{Alert creation}: When \texttt{score > threshold}, an alert is created with a unique ID (\texttt{ALT-XXXXXX}), severity classification (CRITICAL if score $\geq$ 0.92, WARNING if score $\geq$ threshold), and a 24-hour TTL.
    \item \textbf{Email notifications}: When an alert is created, the Lambda publishes to an SNS topic with the \texttt{owner\_id} as a message attribute. Users who have subscribed receive per-user filtered email notifications via SNS filter policies.
    \item \textbf{Incident grouping}: When $\geq$2 active alerts exist within a 5-minute window, they are grouped into an incident (\texttt{INC-XXXXXX}).
\end{itemize}

\subsection{Authentication Flow}

The frontend integrates with AWS Cognito via the \texttt{amazon-cognito-identity-js} SDK:

\begin{enumerate}[leftmargin=2em]
    \item User enters email and password on the \texttt{/login} page
    \item The SDK performs an \textbf{SRP (Secure Remote Password)} handshake with Cognito -- the password is never transmitted in plaintext
    \item Cognito returns JWT tokens (ID Token: 1h, Refresh Token: 30d)
    \item The \texttt{useAuth} context hook provides user state globally
    \item The API client (\texttt{lib/api.ts}) injects the ID token as an \texttt{Authorization} header on every request
    \item The \texttt{ProtectedRoute} component checks authentication and redirects to \texttt{/login} if unauthenticated
    \item On 401 API responses, the user is automatically redirected to \texttt{/login}
\end{enumerate}

\subsection{Theme System}

The dashboard supports \textbf{dark and light themes} with persistent preference:

\begin{itemize}[leftmargin=2em]
    \item Theme state is managed by the \texttt{useTheme} context hook
    \item CSS variables (\texttt{-{}-color-bg-primary}, \texttt{-{}-color-text-primary}, etc.) are toggled by setting \texttt{data-theme} on the document root
    \item Chart colors are dynamically read from CSS variables via the \texttt{chartColors()} utility, ensuring charts respect the current theme
    \item Theme preference is persisted to \texttt{localStorage} and restored on page load
    \item An inline script in \texttt{index.html} applies the theme before first paint to prevent a flash of incorrect colors
\end{itemize}

\subsection{API Integration}

All API calls go through a centralized client (\texttt{lib/api.ts}) that:

\begin{itemize}[leftmargin=2em]
    \item Reads the base URL from \texttt{VITE\_API\_BASE\_URL} (production) or uses \texttt{/api} (development, proxied by Vite)
    \item Automatically injects the Cognito ID token as an \texttt{Authorization} header
    \item Handles 401 responses by redirecting to \texttt{/login}
    \item The custom \texttt{useApi} hook provides \texttt{data}, \texttt{loading}, \texttt{error}, and \texttt{refetch} states, with request cancellation on component unmount to prevent memory leaks
    \item \textbf{Auto-polling}: The \texttt{useApi} hook accepts an optional \texttt{pollInterval} parameter (set to 5 seconds for live pages). Polling pauses automatically when the browser tab is hidden (\texttt{document.hidden} check) and skips ticks when a previous request is still in-flight, preventing request stacking
\end{itemize}

\section{Conclusion}

CloudWatch AI successfully demonstrates a complete, production-grade cloud database application that spans the full stack: from real-time data generation and NoSQL storage, through ML inference and event-driven processing, to interactive visualization and enterprise security.

\begin{infobox}[Implementation Highlights]
\begin{itemize}[leftmargin=1em, itemsep=2pt]
    \item \textbf{Architecture}: Fully serverless on AWS with 13 Lambda functions, 6 DynamoDB tables, SNS email notifications, and event-driven data processing via DynamoDB Streams
    \item \textbf{Database}: DynamoDB's flexible schema elegantly handles 5 different service types with heterogeneous JSON structures, with per-user data isolation via \texttt{owner\_id} partitioning
    \item \textbf{Simulation}: 6 configurable scenarios covering all 5 service types with realistic degradation curves
    \item \textbf{ML Pipeline}: 3 Gradient Boosting models achieving 0.91 precision with $\sim$36-minute lead time
    \item \textbf{Frontend}: React 19 dashboard with auto-polling (5s), email notification toggle, local timezone display, theme system, and authentication
    \item \textbf{Security}: Cognito SRP auth, JWT on all routes, AES-256 encryption, OWASP Top 10 audit passed
    \item \textbf{Testing}: 61 documented test cases across all system layers
    \item \textbf{Cost}: Entire platform runs within AWS Free Tier
\end{itemize}
\end{infobox}

The key contribution is not merely predicting incidents earlier, but producing \textbf{more trustworthy alerts with fewer false alarms}---leading to higher operator trust and faster incident response. This principle of ``precision over recall'' is essential for real-world AIOps adoption.
